CMM with a Twist

  • Tried for a long time to get teams to test
  • Idea: make it in their self-interest
  • Show value immediately and at each stage
  • “Test Certified” Levels…
    • 1: Setup tools – get familiar with testing
    • 2: Write tests for all new code
    • 3: Get all code under test
    • 4: Improve coverage
    • ​

​

This helped teams see value in testing immediately.

​

Teams like organisms, tend to do things in their self-interests.

​

They need to tools and know how, and they will use them if it benefits the individuals.

​

​

How this helped…

  • Fewer emergency pushes
  • Fewer broken builds (compile breaks): smoke tests will remedy this
  • Higher confidence in the product: measure anecdotal responses via surveys
  • Higher rate of change: the team is overcoming fear of change
  • Fewer breakages: 404s, unhandled exceptions, etc.
  • Reduced complexity
  • Higher bug fix rate

​

Test Mercenaries

  • Full timers to help projects on Test Certified
  • Program started by passionate engineers
  • Activities
    • Introduce technologies
    • Refactor code
    • Train individuals on testing and techniques
    • Help team to integrate testing into process
  • Time Boxed Engagement

Clean Build Time

Build speed is a fundamental requirement. If it’s too long, we build up mortgage (as mentioned previously).

​

Can we make the builds so fast that there is no delay? Time waiting is navigating to the results. Or better delivering the results into the devs work-flow (ex IDE).

​

Challenge: some projects have a huge number of build targets. Gmail has 100K!

​

So we need to think about distributed execution, caching, compute avoidance, etc…

​

Chart shows a portion of the spectrum. We have Head and Tail issues. Head build very fast but hard to make faster. Tail explodes into infinity. Required different approaches to solve each part.

Object Caching Wins

Single day:

  • ~500M build actions
  • 94% action cache hit rate
  • 30M cache miss
  • 800 CPU days (just build/test)
  • 66% of actions from automatic builds 
  • Build activity clustered at ends of incrementality, vast majority highly incremental
  • Large number of very small/quick builds and small number of very large/long builds adding noise
  • Cache hit rate key to performance
  • Most build activity from Automated builds

​

We reuse object files build by everyone in the company. So the object cache is always fresh.
